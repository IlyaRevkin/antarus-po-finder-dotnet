# Antarus ПО Finder (.NET / WPF)

[![CI](https://github.com/IlyaRevkin/antarus-po-finder-dotnet/actions/workflows/ci.yml/badge.svg)](https://github.com/IlyaRevkin/antarus-po-finder-dotnet/actions/workflows/ci.yml)

Десктопное приложение для управления базой прошивок и параметров промышленных контроллеров
(ПЛК/УПП/ПЧ) — поиск нужной прошивки по названию шкафа, загрузка новых версий с автоматическим
определением контроллера, хранение параметров ПЧ/УПП, административная панель для настройки
иерархии оборудования.

Это переписывание на .NET 8 / WPF более раннего прототипа на Python (PySide6 + SQLite). Общей
истории коммитов с тем прототипом нет — это независимый репозиторий, архитектура и часть
поведения совпадают намеренно (для паритета функций), часть — сознательно упрощена или изменена
(см. «Отличия и известные ограничения» ниже).

## Архитектура

Три проекта:

| Проект | Назначение |
|---|---|
| `AntarusPoFinder.Core` | Вся бизнес-логика и данные: SQLite (`Database`), доменные модели, иерархия оборудования (`HierarchyService`), поиск (`SearchService`), вся транзакция загрузки прошивки — резолв номера версии, копирование файлов, changelog, вложения, автоопределение контроллера по PSL/KINCO (`FirmwareUploadService`, использует `PslInspector`), файловые операции (`FileSystemHelpers`, `ArchiveExtractor`). Никакого UI — можно было бы переиспользовать под другой интерфейс. |
| `AntarusPoFinder.App` | WPF UI: страницы (Поиск / Осмотр / Модерация тегов / Загрузка / Параметры / Сетевые диски / Тикеты / Настройки), диалоги, темизация. |
| `AntarusPoFinder.Tests` | xUnit-тесты: смоук/юнит-тесты слоя `Core` (`Database`, иерархия, парсинг версий, конфликты синка) плюс полные двухмашинные end-to-end прогоны через реальный `ConfigSyncService`/`TicketSyncService` из `App`. Все фикстуры — временные БД/папки, создаваемые и засеваемые каждым тестом самостоятельно (`TestHelpers/TempDb`, `TempRoot`, `TwoMachines`), никакой тест не зависит от файлов на конкретной машине разработчика. Прогоняются в CI на каждый push/PR в `master` (`.github/workflows/ci.yml`). |

Страницы реализованы как обычные `UserControl` с code-behind (без ViewModel на страницу) —
конструктор принимает `AppServices` (доступ к `Database`/`ConfigService`/`HierarchyService`) и
`IAppHost` (аналог «ссылки на главное окно»: статус-бар, индикатор фоновой работы, перезапуск
таймера синхронизации, обновление сайдбара, смена роли). Диалоги — отдельные `Window` с тем же
подходом.

### Фоновые операции и индикатор занятости

Всё, что ходит на сетевой диск, выполняется **вне потока интерфейса** и **видимо**. Раньше было
наоборот: обход диска, чтение общего конфига, копирование прошивок и создание сотен папок шли прямо
на потоке UI и молча, поэтому окно переставало отвечать — неотличимо от «программа зависла».

Соединение SQLite одно на всё приложение и не потокобезопасно, поэтому просто обернуть операцию
целиком в `Task.Run` нельзя: фоновая синхронизация и любой клик пользователя полезли бы в одно
соединение одновременно. Вместо этого каждая операция «БД → диск» разбита на три отделимые фазы:

| Операция | БД: план | Диск (фон) | БД: запись |
|---|---|---|---|
| Структура папок | `HierarchyService.PlanStructure` | `ApplyStructurePlan` | — |
| Досмотр версий на диске | `PlanFwSync` | `ScanFwDisk` | `ImportFwCandidates` |
| Скан неизвестного | `SnapshotNames` | `ScanUnknownFiles(root, names)` | — |
| Загрузка прошивки | `FirmwareUploadService.Prepare` | `CopyFiles` | `Register` |
| Общий конфиг | — | `ConfigSyncService.CheckForUpdateAsync` / `ExportAsync` | `ApplyAsync` |

Однофазные методы (`EnsureStructure`, `SyncFwFromDisk`, `ScanUnknownFiles(root)`, `Upload`, `Export`)
никуда не делись — остались обёртками над теми же фазами и по-прежнему нормальный способ вызова
там, где блокировать некого (тесты, любой не-UI код). Что дисковые фазы действительно не трогают
базу, проверяется в тестах прямо: они прогоняются на **закрытом** соединении (`BackgroundPhaseSplitTests`).

Видимая часть — `BusyTracker`: единственный индикатор внизу окна, рядом с «Диск: …». Пока идёт
работа, там написано, что именно («Синхронизация прошивок с диском…», «Обновление прошивки: …»,
«Отправка конфига на диск…»), с процентами там, где число шагов известно заранее. Несколько
операций одновременно — показывается последняя начатая, остальные счётчиком «+N». Кнопки, у которых
повторный клик запустил бы вторую такую же работу («Синхронизировать сейчас», «Загрузить»,
«Обновить все»), гаснут на время своей операции; фоновый тик синхронизации, пришедший поверх ещё не
закончившегося предыдущего, просто пропускается.

## Модель данных и иерархия

Прошивки и параметры организованы по 4-уровневой иерархии:

```
ПО/{Группа}/{Подтип}/{Контроллер}/{версия}/файл
Параметры/{Группа}/{Подтип}/{Производитель}/файл
```

- **Группа** — тип шкафа верхнего уровня (НГР — насосная, ПЖ — пожарное водоснабжение, ТГР —
  тепловая); у каждой свой числовой префикс версии.
- **Подтип** — КПЧ/ВЗУ/КНС/ПП/ХП/FD и т.д.; подтип `—` означает «подтипа нет» (папка
  пропускается, актуально для ТГР).
- **Контроллер** — модель (SMH4/SMH5/KINCO/PIXEL2/PIXEL), у каждой — список **модификаций**
  (например PIXEL2-1020, PIXEL2-1321) со своим `hw_version`.
- **Версия прошивки** — строка формата `eq_prefix.sub_prefix.hw_version.sw_version.ГГГГММДД_ЧЧММ`,
  например `2.1.42.1.20260714_1530`. `sw_version` — счётчик, автоматически увеличивается на
  единицу при каждой новой загрузке для той же тройки (подтип, контроллер, hw_version).

Обе стороны (диск и БД) должны совпадать: `HierarchyService.EnsureStructure` создаёт на диске
недостающие папки по данным из БД, `SyncFwFromDisk` — наоборот, добавляет в БД записи о версиях,
физически найденных на диске, но отсутствующих в базе.

**Досмотр диска идёт сам, тем же периодическим тиком, что и остальная синхронизация.** Раньше он
вызывался только внутри применения нового общего конфига, а конфиг **отправляет** лишь
администратор и по умолчанию не отправляет вовсе (`config_push_interval_min = 0`) — то есть у
большинства машин обход диска не запускался никогда. Отсюда и жалоба «позагружал прошивки на компе
коллеги, у себя не вижу»: файлы на диске были, показать их было некому. Что доезжает вместе с
версией:

- **описание и типы пуска** — из `CHANGELOG.md` в папке версии;
- **теги** — оттуда же (разделитель `; `, поэтому многословный тег переживает дорогу целиком) и
  заодно попадают в справочник тегов, иначе версия находилась бы только по названию папки;
- **нестандартные (ОПЦ) версии** — папка «ОПЦ» общая на все контроллеры подтипа и лежит рядом с
  ними, поэтому досмотр не открывал её вообще. Контроллер выводится из `hw_version` в номере
  версии (незнакомый или неоднозначный `hw` — версия пропускается: завести её не тому контроллеру
  хуже, чем не завести вовсе), а номер заявки и заводской SN восстанавливаются из имени файла —
  больше их на диске нигде нет.

## Возможности по страницам

### Поиск

Токенизированный полнотекстовый поиск по иерархии: запрос нормализуется (разделители `,;-/\`
схлопываются в пробелы, приводится к верхнему регистру), затем каждая прошивка в БД получает
очки — по числу совпавших частей иерархии (группа/подтип/папка-подтипа/контроллер, только если
длина ≥ 2 символов), плюс ×2 за каждое совпавшее слово из тегов записи и столько же за **тип
пуска**, отмеченный галочкой при загрузке версии (`launch_types`). Тип пуска сравнивается **целым
значением**, а не подстрокой, и независимо от галочки «точное совпадение слова»: список закрытый
(УПП/ПП/ПЧ/КПЧ/Отсутствует) и почти каждое значение в нём — подстрока другого, поэтому запрос
«НГР ПЧ» иначе поднимал заодно шкафы с КПЧ. Из всех версий одного и того же (подтип, контроллер)
в выдачу попадает только версия с максимальным очком, финальный список сортируется по убыванию
очков.

Тег — это **целый тег**, а не слово из него. Тегом бывает полное название шкафа («шкаф управления
пожарными насосами Антарус ПЖ-ПП-2-…»), а строка тегов в БД исторически разделяется пробелом, и
такой тег рассыпался на слова при первом же сохранении. Пробел внутри тега кодируется
(`TagString`), формат строки не поменялся — старые однословные теги читаются как читались. Из этого
следуют два правила выдачи:

- запрос, совпавший с тегом **целиком**, весит больше любого набора отдельных слов;
- галочка **«точное совпадение слова»** означает ещё и «**все** слова запроса, а не любое из них».
  Раньше хватало одного совпавшего слова: нужная версия шла первой, а следом — всё, что случайно
  совпало словом «шкаф». Наладчик, поставивший галочку и вбивший точное название шкафа, получает
  ровно ту прошивку, которой этот тег проставили.

**Частота выбора.** Приложение запоминает, какую версию оператор взял по такому же запросу
(`fw_search_usage`: открыл проект/файл, залил в контроллер или скачал локально — просмотр карт,
инструкций и истории не в счёт). В следующий раз по тому же запросу эта версия идёт выше среди
**равно подходящих**, а на карточке написано «по этому запросу выбирали N раз» — иначе подъём
выглядел бы необъяснимым. Вклад частоты ограничен (`MaxUsageBonus`): сто открытий прошивки другого
шкафа не должны вытаскивать её в чужую выдачу — это подсказка среди подходящего, а не замена
релевантности. Таблица локальная: в общий конфиг не выгружается и с других машин не приезжает.

**Фильтры** — отдельная кнопка справа от строки поиска, свёрнута по умолчанию (в строке и так три
кнопки, а фильтры нужны не каждый раз): тип шкафа, подтип, контроллер, тип пуска, тег. Фильтр —
это «показывай только такое», а не «подними повыше»: он применяется **до** подсчёта очков. Поле
тега — с поиском по списку: набранное сужает выпадающий список (у наладчика он длинный), и
предлагаются только теги, реально проставленные на версиях. Пустой запрос **с** фильтрами —
осмысленное «покажи всё такое» (все прошивки НГР на SMH5, все с типом пуска ПЧ). Свёрнутая панель
фильтры не отменяет — поэтому на кнопке стоит точка, когда что-то выбрано, а «Сбросить» очищает и
запрос, и фильтры.

**Снимок для поиска.** Раньше каждый поиск заново вычитывал все `fw_versions` с тремя JOIN'ами — на
каждое нажатие «Найти», на каждое переключение режима, на каждый молчаливый повтор запроса (возврат
на вкладку, тик синхронизации, закрытие диалога тегов). Теперь снимок читается один раз и
переиспользуется, пока данные не менялись. Ревизия данных поднимается в `Exec`/`ExecuteNonQuery` —
то есть в единственных двух местах, через которые проходит любая запись: про инвалидацию нельзя
забыть, добавив новый метод записи. Плюс индексы по колонкам, по которым реально ищут
(`subtype_id, controller_id`, `version_raw`, резервации, параметры) — без них проверка «какие версии
уже есть у этой пары подтип/контроллер» была полным перебором таблицы на каждую папку контроллера.

Каждый результат — карточка. Кнопок со временем стало слишком много для одного ряда, поэтому в
основном ряду остались только те, ради которых карточку и открывают, а всё остальное убрано в меню
**«Ещё»** (всплывающий список под кнопкой).

Под заголовком карточки — строка **«Файлы:»**: что реально лежит рядом с версией (`LFS ✓ · PSL — ·
HMI ✓ · параметры ✓ · карта ВВ ✓ …`). Раньше «есть ли LFS» приходилось выяснять, открыв меню и не
найдя там строки. Прочерк у LFS/PSL показывается только для Segnetics-проектов (`SegneticsProject`)
— у шкафа на KINCO таких файлов не бывает в принципе, и «PSL —» читалось бы как потерянный файл.
Пока идёт обход папки версии, в строке стоит «проверяем папку версии…» — сам обход идёт в фоне
(карточки рисуются сразу, иначе выдача подвисала на каждый результат, читая сетевую папку).

Описания версии на карточке **нет**: это несколько строк changelog'а на каждой карточке выдачи, без
предела по объёму. Читается там, где оно осмысленно — «Ещё» → «История версий», где заодно видно,
чем версии отличаются друг от друга.

Основной ряд:

- **Открыть прошивку (LFS)** / **Открыть проект (PSL)** и (если есть что открывать) **Открыть HMI
  проект** — открывают файл **только из локального кэша**, никогда не трогают сеть. Какой именно
  файл открыть, решается в одном порядке для любого вендора, а не по захардкоженному списку
  расширений: отдельно загруженный HMI-проект → **исполняемый файл, явно указанный оператором**
  (`ExecutableHint` / `HmiExecutableHint`, относительный путь — файл может лежать во вложенной
  папке) → детект по KINCO-расширениям как фоллбэк для старых записей без подсказок. Так же
  открывается и проект любого другого вендора, у которого программа ПЛК и файл панели лежат в одной
  папке.

  **Расширение всегда написано на кнопке, в скобках**, и по ней сразу видно, что откроется.
  Приоритет у `.lfs` — это файл, который заливают в контроллер; `.psl` — исходник, его открывают,
  когда нужно править. Второй файл пары уезжает в «Ещё»: есть оба — на кнопке LFS, в «Ещё» PSL;
  есть только один — он на кнопке, а второй остаётся в «Ещё» **неактивным пунктом с причиной**
  (пустое место в меню неотличимо от «я не туда посмотрел»). Ситуации «есть PSL, нет LFS» после
  допиливания лоадера быть не должно.
- **Параметры** — только если для записи есть файл параметров.
- **Загрузить в ПЛК** — загрузка через лоадер (см. «Лоадер» ниже). Если `.lfs` найден, кнопка
  подписана **«Загрузить в ПЛК (LFS)»** и файл подставится в диалог сам.

Меню «Ещё» разбито на разделы: **Файлы версии** (открыть папку, второй файл пары LFS/PSL, обновить
локальную копию с диска), **Документация** (карта in/out, карта modbus,
инструкции), **Версия** (история версий, теги и описание — только администратору). Пункты, для
которых файла нет, **не исчезают, а становятся неактивными** с объяснением в подсказке: пропавшая
строка «Открыть прошивку (LFS)» не отличима от «я не туда посмотрел». LFS/PSL признаются существующими
только по папкам **самой** версии — раньше в счёт шли и соседние версии из локального кэша, из-за
чего карточка писала «LFS ✓» там, где его не выкладывали, а в контроллер подставился бы `.lfs`
чужой версии.

**История версий** (двойной клик по строке открывает папку версии) помечает **одну** версию как
текущую, а не всю историю сразу: самая свежая живая — «Текущая» (жирным), остальные — «Заменена»,
откатанные — «Откатана». Версии с другим `hw_version` — это другое аппаратное исполнение шкафа, и
свежая версия под иное железо ничего не заменяет: самая новая версия каждого hw помечается
«Текущая (HW n)», чтобы наладчик со старой платой видел, что актуально ему.

**Локальная копия подтягивается сама.** Отдельной кнопки «Синхронизировать» на карточке больше нет:
версии из выдачи, которых ещё нет в `%LocalAppData%\AntarusPOFinder\firmware\<имя>\<версия>`,
копируются с сетевого диска в фоне, а карточка показывает состояние («синхронизируем…» → «✓
локальная копия обновлена» / текст ошибки); общий ход при этом виден и внизу окна, в индикаторе
занятости — карточка может быть уже прокручена за экран. Копирование последовательное (сетевой диск
компании регулярно отваливается, десяток параллельных копий делу не помогает) и прекращается, если
оператор успел запустить новый поиск. За одну выдачу тянется не больше 10 версий — сколько всего нашлось и
сколько взято, видно в строке статуса, остальные копируются вручную через меню «Ещё». Выключается в
Настройки → Общие → «Поиск».

Отдельно на панели инструментов: смена/очистка папки осмотра, упрощённый скан документа (запуск
приложения «Сканер Windows» + сравнение содержимого стандартных папок вывода до/после — без
прямого управления сканером через WIA, см. «Известные ограничения»), и загрузка фото с телефона
по QR-коду (см. ниже).

### Лоадер (загрузка в контроллер) — заготовка

Кнопка **«Загрузить в ПЛК»** в карточке поиска открывает диалог загрузки: параметры (**форматировать
память**, **обновить ядро**, порт/адрес контроллера, файл — подставляется найденный `.lfs`),
прогресс-бар, лог с цветными строками, кнопки «Остановить», «Сохранить лог», «Рабочая папка».

**Реального лоадера пока нет** — подключена заготовка (`StubFirmwareLoaderBackend`): она проходит те
же стадии и пишет тот же лог, но ничего не собирает и не заливает. Это видно всё время: баннер в
диалоге, префикс `[заглушка]` в каждой строке лога, а вместо `.lfs` в результат кладётся
файл-памятка — результат заготовки нельзя перепутать с настоящей сборкой, и на диск он не
публикуется.

Всё делается **на локальной машине**: приложение не клиент-серверное, поэтому исходник сначала
копируется в локальную рабочую область (`%LocalAppData%\AntarusPOFinder\loader\<дата>_<версия>\`,
подпапки `src` / `out`, лог запуска рядом), лоадер работает только с локальными путями, и лишь
успешный результат сборки публикуется в папку версии на диске — **докладыванием** файлов, не заменой
папки целиком. Рабочие области старше 7 дней подчищаются автоматически.

Подключение настоящего лоадера — реализовать `IFirmwareLoaderBackend` и вернуть его из
`FirmwareLoaderFactory.Create`; больше ничего менять не нужно. Полный контракт, требования и
открытые вопросы к разработчику лоадера — [`docs/loader-integration.md`](docs/loader-integration.md).
Путь к исполняемому файлу лоадера задаётся в Настройки → Общие → «Лоадер».

### Загрузка

Форма загрузки новой версии прошивки. Вся логика транзакции загрузки (резолв номера версии,
копирование файла/папки и вложений, запись `CHANGELOG.md`, тегирование, исполнение резервации)
живёт в `AntarusPoFinder.Core.Services.FirmwareUploadService` — чистый метод
`Upload(request) → result`, без зависимостей от WPF, покрытый юнит-тестами. `UploadView`
(code-behind) собирает `FirmwareUploadRequest` из полей формы, вызывает сервис и показывает
результат (успех/ошибка/предупреждения); два бывших модальных диалога-подтверждения (неизвестное
расширение, папка версии уже существует) сервис возвращает как `NeedsConfirmation` — `UploadView`
переспрашивает пользователя и повторяет вызов с подтверждённым флагом. Сам `Upload` — обёртка над
тремя фазами `Prepare` → `CopyFiles` → `Register` (см. «Фоновые операции» выше): `UploadView`
вызывает их по отдельности, чтобы копирование гигабайтов на сетевой диск шло в фоновом потоке с
прогрессом внизу окна. Оба подтверждения спрашиваются между `Prepare` и `CopyFiles`, то есть до
того, как на диск записан хоть один байт — отказ пользователя не оставляет следов. Автодетект контроллера по
`.psl`/KINCO-расширению (см. п. 2 ниже) тоже вызывает методы `FirmwareUploadService`, а не хранит
эту логику в code-behind. Четыре поля вложений (Карта in/out, Карта modbus, Инструкция,
HMI-проект) используют общий UI-хелпер `FilePickerRow` (`AntarusPoFinder.App/Views`) для кнопок
«Файл…»/«Папка…»/«Очистить» вместо дублирования диалогового кода в каждом поле.

1. Выбор Группы → Подтипа → Модификации контроллера (единый список модификаций всех
   контроллеров, не сгруппированный).
2. **Автоопределение контроллера по содержимому `.psl`-файла** (см. `PslInspector` в `Core`):
   в бинарном файле проекта Segnetics SMLogix ищутся JSON-фрагмент `"key":"..."` / `"owner":"..."`
   или XML-тег `<device>...</device>`; извлечённый идентификатор (например `PIXEL2-1020-70-4`)
   разбирается регулярным выражением на модель (`PIXEL2`) и модификацию (`1020`), и если в
   справочнике находится точное совпадение — комбобокс переключается сам с пометкой
   «🔍 автоопределено из файла». Никогда не блокирует загрузку и не выбирает ничего насильно.
3. Два независимых переключателя нестандартного шкафа — **«ОПЦ заявка»** (раскрывает поле № заявки,
   суффикс `_(номер)` в имени файла) и **«ОПЦ SN»** (раскрывает поле серийного номера шкафа, суффикс
   `_SNxxxxx`). Наладчик/программист включает и заполняет только то, что реально есть для конкретного
   шкафа — они не обязаны идти парой. Если включён хотя бы один из двух, целевая папка на диске
   становится `ОПЦ` вместо имени контроллера.
4. Проверка **белого списка расширений** — если расширение загружаемого файла не входит в
   список разрешённых (настраивается в Настройках), показывается предупреждение с
   подтверждением; список пуст ⇒ проверка отключена целиком. Проверка применяется только к
   одиночным файлам, не к папкам.
5. Живой предпросмотр итогового пути и имени файла на диске, пересчитывается на каждое
   изменение формы (использует «следующий» `sw_version`, реально ещё не занятый).
6. Обязательное поле «Изменения в этой версии» — текст идёт и в поле `description` в БД, и в
   файл `CHANGELOG.md`, который создаётся (перезаписывается) в папке версии в формате:
   ```
   # {версия}
   Дата: {ГГГГММДД_ЧЧММ}
   Тип пуска: {через запятую}
   Теги: {через «; »}

   {текст изменений}
   ```
   Теги — через `; `, а не через пробел: тег бывает из нескольких слов (полное название шкафа).
   Строки «Теги:» может не быть — это файл старого формата, разбор к этому готов.
7. **Имя файла содержит только числовой номер версии** (`eq.sub.hw.sw[.дата_время]`), плюс
   опционально суффиксы `_(заявка)`/`_SNxxxxx` — например `2.1.042.0001.20260715_1824.PSL`. Группа/
   подтип/контроллер в имя файла больше не подставляются (эта правка — более позднего раунда, чем
   пример ниже мог предполагать): они и так видны из пути на диске (`ПО/Группа/Подтип/Контроллер/...`)
   и разбираются обратно из самого номера версии (`FwVersionNumber.Parse`) — вместо этого при
   загрузке они автоматически добавляются как теги (наравне с тегами, которые ввёл оператор), так
   что поиск по тегам продолжает находить файл по тем же словам. Ранее одиночные `.psl`-файлы также
   всегда получали суффикс `_0` (была отсылка к ожиданиям firmware-тулчейна Segnetics) — этот
   суффикс убран; если это в реальности ломает тулчейн — стоит перепроверить перед следующим релизом
   (см. README «Отличия и известные ограничения» и итоговый отчёт агента для контекста).
8. **HMI-проект** — отдельный чекбокс «Добавить HMI-проект» раскрывает свою собственную зону
   drag&drop (с независимыми кнопками «Файл…»/«Папка…»/«Очистить»), для контроллеров, где HMI —
   отдельный проект со своими файлами (например Segnetics, `.fsprj`), не часть основного файла
   прошивки. HMI **не загружается отдельно** — он всегда прикладывается к версии прошивки ПЛК:
   если заполнена только HMI-зона, а основная (файл/папка прошивки) пустая, «Загрузить прошивку»
   покажет явное объяснение вместо попытки загрузить HMI как саму прошивку.

   **Панель и программа ПЛК обновляются независимо.** У одного шкафа их правят порознь: сегодня
   поправили логику, панель не трогали. Версия, загруженная **без** панели, подхватывает HMI-проект
   последней живой версии этого же шкафа (тот же подтип + контроллер, см.
   `Database.GetLatestHmiForFirmware`) — раньше она оставалась просто без панели, и на карточке
   кнопки «Открыть HMI проект» не было вовсе. Файлы при этом **не копируются второй раз**: путь
   указывает на ту же папку проекта, что и у предыдущей версии. Карточка честно пишет, откуда
   панель: «Открыть HMI проект **(от 3.0.005.0001)**» — не делать же вид, что она собрана вместе с
   этой версией. Загрузка со своей панелью, понятно, ничего не наследует.
9. **Доп. файлы** (необязательные, каждый со своим полем/кнопками «Файл…»/«Папка…»/«Очистить») —
   Карта in/out (копируется в подпапку контроллера «Карта ВВ»), Карта modbus («Карта Modbus»),
   Инструкция («Инструкция»).
10. **Несколько подтипов шкафа для одной прошивки** — кнопка «Ещё подтипы…» рядом с комбобоксом
   подтипа. Основной подтип задаёт номер версии, папку на диске и резерв номера; каждому
   дополнительно выбранному подтипу заводится своя запись в `fw_versions` с тем же номером и тем же
   `disk_path`, а в папку его контроллера кладётся **ярлык** (`.lnk`) на папку версии. Файлы на
   диске не дублируются — прошивка, подходящая пяти подтипам, занимает место один раз, но находится
   поиском под каждым из них (`FirmwareUploadRequest.ExtraSubtypes`, `IShortcutCreator`). Если ярлык
   создать не удалось (сетевая папка, отсутствующий `WScript.Shell`), загрузка не отменяется —
   выводится предупреждение, записи уже заведены.
11. **Набор файлов вместо папки** — в зону drag&drop (и в диалоге «Файл…») основной прошивки и HMI
   можно перетащить/выбрать сразу несколько файлов. Они складываются в одну временную папку
   (`DropStagingService`), и дальше это обычный сценарий «загрузили папку»: те же проверки, тот же
   выбор исполняемого файла, то же копирование целиком. Временная папка удаляется после загрузки и
   при очистке формы. Раньше из набора молча брался первый файл, остальные пропадали без сообщения.
12. **Какой файл в папке «исполняемый»** — если загружается папка, приложение спрашивает, какой файл
   внутри открывать по кнопке в поиске (`PickFileDialog`, двойной клик заходит в подпапки).
   Единственный файл с «родным» расширением во всём дереве подставляется сам
   (`ExecutableHintResolver.AutoDetect`). Подсказку можно переназначить позже — см. «Правка
   прошивки» ниже.
13. **Тип пуска** (обязательный) — чекбоксы, можно выбрать несколько. Пятый — **«Отсутствует»**,
   для шкафов без типа пуска: при его выборе остальные снимаются и блокируются, чтобы не получилось
   записи «отсутствует + прямой» одновременно.
14. **Откат версии** — помечает последнюю активную версию для выбранной тройки (подтип,
   контроллер, hw_version) как откатанную (`status = rolled_back`); следующая загрузка получит
   тот же самый `sw_version` заново. Файлы на диске никогда не удаляются автоматически — это
   единственный механизм отмены.

#### Правка уже загруженной прошивки

Диалог `EditFirmwareDialog` (открывается из карточки поиска, вкладки «Новые версии» и
Настройки → Прошивки) правит не только описание/теги/типы пуска:

- **Доп. файлы** — Карта in/out, Карта modbus, Инструкция и HMI-проект можно **доложить к уже
  загруженной версии**, если файл прислали позже: `FirmwareAttachmentsService` кладёт их ровно
  туда же, куда их кладёт загрузка новой версии, перезаливать версию целиком не нужно. Пустое поле
  убирает ссылку, файлы на диске остаются (папки общие для всех версий контроллера); при замене
  HMI старая папка проекта сносится, а не мержится.
- **Исполняемые файлы** — переназначить, какой файл открывается по кнопкам «Открыть прошивку ПЛК»
  и «Открыть HMI проект» (двойной клик в диалоге выбора заходит в подпапки). Если отдельной папки
  HMI-проекта у версии нет, файл панели выбирается прямо в папке версии — так поддерживается любой
  проект, где программа ПЛК и панель лежат рядом, без отдельного функционала под конкретного
  вендора.

### Параметры

Загрузка и просмотр файлов параметров ПЧ/УПП, привязанных к (Группа, Подтип, Производитель).
Удаление записи — мягкое (`archived = 1` в БД), файл на диске никогда не удаляется автоматически.
Кнопка «Ещё подтипы…» работает так же, как в «Загрузке ПО»: файл копируется на диск один раз, а
остальным выбранным подтипам заводится запись с тем же `disk_path` и кладётся ярлык
(`ParamFileLinkService`).

### Осмотр

Приём фото/сканов с телефона (по QR-коду, см. ниже) и файловый менеджер папки осмотра с
предпросмотром. Предпросмотр изображения автоматически масштабируется под размер панели (вписан
пропорционально, `Viewbox`) и поддерживает минимальное редактирование: поворот на 90°/270°
(`⟲`/`⟳`, без потери качества — каждый поворот считается заново от исходного файла, а не от уже
повёрнутой картинки), с сохранением либо поверх исходного файла (с подтверждением — папка осмотра
рабочая, не архивная), либо отдельной копией. Обрезка (crop) не реализована — сознательно оставлено
за рамками минимального набора правок.

Рядом с QR-кодом — заметное напоминание, что телефон должен быть в той же Wi-Fi/локальной сети, что
и этот компьютер (иначе ссылка из QR не откроется), и кнопка «Как подключить папку с телефона» с
пошаговой инструкцией (Android/iOS) для постоянного подключения папки осмотра как сетевой папки
(SMB) через файловый менеджер телефона — в отличие от QR-загрузки (которая всё равно требует каждый
раз открыть ссылку и нажать «Отправить»), после такого подключения любое фото, сохранённое в эту
папку с телефона, сразу считается «залитым», без отдельного шага. Инструкция пытается сама
определить и показать реальный сетевой (`\\сервер\...`) адрес папки осмотра, если она лежит на
подключённом сетевом диске.

### Тикеты

Баг-репорты и предложения по самой программе — отдельная страница в сайдбаре, видна всем ролям, но
с разным набором действий:

- **Наладчик / Программист** — создают тикеты (тип: баг/предложение/другое + текст) и видят только
  **свои** — привязка идёт по имени пользователя Windows (`Environment.UserName`), так как роли в
  приложении общие/парольные, а не персональные учётки.
- **Администратор** — видит тикеты всех пользователей со всех машин и может менять статус (Открыт →
  В работе → Закрыт, либо переоткрыть).

Хранение — локальная таблица `tickets` в `po_finder.db` на каждой машине, синхронизация между
компьютерами — отдельный от общего конфига (`ConfigSyncService`) канал: каждое действие (создание
тикета, смена статуса) сразу пишется отдельным JSON-файлом-событием в
`{диск}\Конфиг\tickets\` (maildir-подобный append-only лог, см. `TicketSyncService`), а не как часть
единого снимка `po_finder_config.json`. Так несколько человек, создающих тикеты с разных машин
одновременно, никогда не затирают тикеты друг друга — в отличие от целиком перезаписываемого
снимка конфига, где это было бы гонкой. Каждая машина независимо подтягивает ещё не увиденные файлы
при каждом открытии страницы «Тикеты»; смена статуса разрешается по принципу «побеждает более
новое событие» (по временной метке), так что применение событий в любом порядке или на нескольких
машинах сходится к одному и тому же результату. События, которые не удалось записать на диск сразу
(сеть недоступна), остаются в локальной очереди (`ticket_outbox`) и досылаются при следующем
открытии страницы. Если при открытии страницы что-то не удалось отправить/принять, это больше не
тихая ошибка — статус-бар/история уведомлений сообщают «Синхронизация тикетов: не удалось обработать
файлов: N», а при следующем успешном открытии — «Синхронизация тикетов восстановлена» (уведомление
показывается один раз на переход в проблемное/нормальное состояние, а не на каждое открытие
страницы, пока диск недоступен).

### Сетевые диски

Отдельная страница (не вкладка Настроек — доступна всем ролям, так как каждый настраивает свой
компьютер сам, и эти пути/интервалы принципиально локальны, см. ниже):

- **Путь к диску** (WebDAV / сетевая папка) — корень, под которым живут `ПО/`, `Параметры/` и
  служебная папка `Конфиг/` (справочник, теги, резервы номеров, тикеты, обмен конфигом между
  машинами).
- **Второй диск** — зарезервирован под шкафы/электросхемы, см. «Отличия и известные ограничения».
- **Папка осмотра** — куда попадают сканы/фото по QR и файлы, накопленные во время осмотра.
- **Получение обновлений конфига** — интервал (в минутах) автоматической подтяжки изменений
  справочника/тегов/резервов, сделанных на других машинах (0 = только вручную, кнопка
  «Синхронизировать сейчас» всегда доступна). Конфликты (одна и та же запись независимо изменена
  на двух машинах с последнего согласованного состояния) не применяются автоматически — видны и
  решаются через «Показать и решить».
- **Отправка изменений на диск** — то же самое в обратную сторону, свой отдельный интервал и
  кнопка «Отправить сейчас».

**Изменения общего справочника отправляются сразу, не дожидаясь интервала.** Тип и подтип шкафа,
контроллер, модификация, производитель ПЧ/УПП, расширение файла — после любой такой правки
приложение само делает «сначала забрать чужое, потом отдать своё» (`PushCatalogChange`). Отправка
по-прежнему разрешена только администратору, а её интервал по умолчанию выключен (0) — из-за чего
добавленный производитель физически не доезжал до коллег, пока администратор отдельно не зайдёт на
эту страницу и не нажмёт «Отправить сейчас».

### Настройки

Восемь вкладок:

- **Общие** — то, что реально требует прав администратора и НЕ синхронизируется по сети (пути к
  дискам и интервалы синхронизации переехали на страницу «Сетевые диски» выше): смена роли,
  пароли ролей (Администратор/Программист — у Наладчика пароля нет, это роль по умолчанию), вход
  по AD (домен, группы ролей, способ проверки пароля — LDAP/HTTP/оба, обязательный вход при
  запуске приложения — см. «Роли и доступ» ниже), «хранить архивы после извлечения», действие при
  закрытии окна (закрыть/свернуть в трей), автозапуск при входе в Windows, обновление самого
  приложения (см. «Обновление приложения» ниже).
- **Иерархия** — CRUD для групп/подтипов/контроллеров/модификаций/производителей параметров
  (одна строка таблицы — один подтип; тип, у которого подтипов не осталось вовсе, показывается
  строкой «(нет подтипов)» — иначе он невидим здесь, но по-прежнему предлагается в «Загрузке ПО»,
  и удалить его из интерфейса нечем),
  список разрешённых расширений, кнопки «Пересоздать структуру диска» / «Синхронизировать
  прошивки с диска» / «Сканировать неизвестные файлы» (папки на диске, не соответствующие
  известной иерархии — по каждой можно выбрать: перенести в «Неизвестное» или удалить
  безвозвратно).
- **Прошивки** — таблица всех загруженных версий с фильтром, редактирование описания/тегов/типа
  пуска, дублирование записи, откат версии, **удаление версии** (безвозвратно с этого компьютера —
  помечается tombstone-удалением, а не просто стирается из базы, специально чтобы удаление
  корректно долетело и до остальных машин при следующей синхронизации конфига, а не «воскресло»
  там снова, см. `Database.TombstoneFwVersion`) прямо по выбранной строке.
- **Модерация** — версии, ещё не выведенные из модерации (см. страницу «Модерация тегов» в
  сайдбаре — то же самое, урезанный вид для наладчика/программиста).
- **Резервация номеров** — активные резервы номера версии (см. «Зарезервировать номер версии» в
  Загрузке).
- **Теги** — список всех тегов, встречавшихся в прошивках/параметрах.
- **Быстрый доступ** — настраиваемые ярлыки быстрого запуска в сайдбаре (с иконкой,
  извлечённой из самого exe/lnk-файла).
- **Пользователи** — собственный ростер приложения (AD-логин, роль, первый/последний вход) для
  всех, кто хоть раз вошёл по AD-логину/паролю и не попал в одну из настроенных AD-групп ролей
  (см. ниже); администратор может сменить роль записи в ростере вручную.

#### Роли и доступ

| Роль | Поиск | Осмотр | Модерация тегов | Загрузка | Параметры | Настройки* | Сетевые диски | Тикеты |
|---|:-:|:-:|:-:|:-:|:-:|:-:|:-:|:-:|
| Наладчик | ✓ | ✓ | ✓ | | ✓ | урезано | ✓ | ✓ |
| Программист | ✓ | | | ✓ | ✓ | урезано | ✓ | ✓ |
| Администратор | ✓ | ✓ | ✓ | ✓ | ✓ | полностью | ✓ | ✓ |

\* Наладчику/программисту в Настройках видна только вкладка «Быстрый доступ» и срок повторного
входа по AD лично для себя — смена ролей/паролей, AD-группы, способ проверки, вкладки
Иерархия/Прошивки/Модерация/Резервация/Теги/Пользователи скрыты целиком.

Три равноправных способа сменить роль в диалоге «Сменить роль» (доступен из сайдбара):

1. **Пароль роли** — выбрать роль из списка и ввести её пароль (общий на роль, не персональная
   учётка). У Администратора пароль обязателен (по умолчанию `12345`, меняется в Настройках →
   Общие), у Программиста — опционален (пусто в настройках = без пароля), у Наладчика пароля нет
   вообще.
2. **Вход по AD, через настроенную группу** — ввести домен/логин/пароль AD-учётки; если логин
   входит в одну из групп, указанных в Настройках → Общие → AD (по одной группе на роль), роль
   выдаётся сразу, без пароля роли. Не требует, чтобы сама Windows-сессия шла под этой учёткой.
3. **Вход по AD, без настроенной группы** — тот же домен/логин/пароль, но если группы для ролей не
   заданы (или логин ни в одну из них не входит): роль берётся из собственного ростера приложения
   (вкладка Настройки → Пользователи) — уже известный логин получает сохранённую там роль, самый
   первый вход этого логина автоматически становится «Наладчиком» (администратор может сменить
   роль в ростере вручную). Ни группы, ни ростер не требуют домена/IT-поддержки для работы — при
   их отсутствии просто ничего не ломается, доступен обычный вход паролем роли.

Вся настройка AD (домен, группы, способ проверки, обязательный вход при запуске) — **локальная для
каждого компьютера**, не разъезжается по сети вместе с остальным конфигом: администратору нужно
задать её один раз на каждой машине отдельно. Опционально можно включить «Требовать вход по AD при
запуске» — тогда при каждом старте приложения (если нет ещё действующего запомненного входа на этом
компьютере, срок настраивается) появляется окно входа вместо немедленного показа главного окна;
пароль администратора остаётся аварийным способом входа, чтобы не оказаться заблокированным при
проблеме с доменом.

#### Загрузка фото с телефона

Кнопка «Фото» на странице Поиска поднимает мини-HTTP-сервер (на порту из настроек, по
умолчанию 9876) и показывает QR-код со ссылкой — телефон в той же сети открывает простую
HTML-форму загрузки и отправляет фото прямо в папку осмотра. Сервер реализован на «сыром»
`TcpListener` с самодельным разбором HTTP-запроса и multipart-тела, **намеренно не через**
`HttpListener`: у Windows `HttpListener` (HTTP.SYS) требует прав администратора или
предварительной регистрации URL (`netsh http add urlacl`) для любого адреса кроме localhost —
это сломало бы функцию для обычного пользователя без прав администратора.

#### Обновление приложения

Источник обновлений — по одному правилу везде (плашка при запуске, ручная проверка в Настройках):
если в Настройки → Общие → «Обновление приложения» указана сетевая папка — релизы ищутся в ней;
если папка пуста — используется публичный GitHub-репозиторий
([IlyaRevkin/antarus-po-finder-dotnet](https://github.com/IlyaRevkin/antarus-po-finder-dotnet)).
Источник, из которого взято обновление, всегда написан в тексте плашки/статуса, чтобы было видно,
что именно проверялось.

- **Папка**: файлы вида `AntarusPoFinder-{версия}.exe`, например `AntarusPoFinder-1.1.0.exe` — по
  одному файлу на выпущенную версию, старые версии не удаляются.
- **GitHub**: релизы репозитория (`gh release list`), версия берётся из тега (`v1.1.0` →
  `1.1.0`), файл — первый `.exe`-ассет релиза.

В обоих случаях версия сравнивается с текущей (читается из атрибутов сборки, `<Version>` в
`AntarusPoFinder.App.csproj`). Выпадающий список со всеми найденными версиями и кнопка
«Установить выбранную» позволяют поставить любую версию, в том числе более старую — это и есть
откат приложения (без переустановки вручную с чужого компьютера). Установка одинакова во всех
случаях: нужный `.exe` (локальный или скачанный с GitHub) копируется рядом с текущим, приложение
закрывается, короткий `.bat`-скрипт в `%TEMP%` дожидается завершения процесса, подменяет `.exe`
на выбранную версию и перезапускает приложение.

При запуске приложение само проверяет обновления (через ~2.5 с) и, если найдена более новая
версия, показывает плашку сверху окна с кнопками «Обновить сейчас» / «Позже». Галочка
«Устанавливать найденные обновления автоматически при запуске» в тех же настройках включает
полностью автоматическую установку без подтверждения.

Чтобы выпустить новую версию:

1. Поднять `<Version>` в `AntarusPoFinder.App.csproj`.
2. `installer/build.ps1` (без `-Version` — читает его из csproj, см. шаг 1) публикует портативный
   self-contained `.exe` и собирает вокруг него MSI-инсталлятор (см. `installer/Package.wxs`) одной
   командой:
   ```powershell
   powershell -File installer/build.ps1
   ```
   Результат — `installer/AntarusPoFinder-{версия}.exe` (портативный, без установки) и
   `installer/AntarusPoFinder-{версия}-setup.msi` (устанавливает в `%LocalAppData%`, добавляет
   ярлыки, поддерживает апгрейд/переустановку той же или более новой версии).
3. Переименовать/скопировать exe в формат `AntarusPoFinder-{версия}.exe` (уже в этом виде из шага 2)
   и либо положить оба файла в папку обновлений на сетевом диске рядом с предыдущими версиями, либо
   опубликовать GitHub-релиз с **обоими** файлами:
   `gh release create v{версия} AntarusPoFinder-{версия}.exe AntarusPoFinder-{версия}-setup.msi`.
   Приложение само проверяет обновления только через `.exe`-ассет (портативная подмена файла, см.
   выше) — MSI в релизе нужен для тех, кто ставит/переустанавливает приложение впервые или через
   инсталлятор, а не через самообновление.

## Сборка и запуск

Требуется .NET 8 SDK.

```bash
dotnet build                 # долг / debug-сборка всего решения
dotnet test                  # смоук-тесты Core
dotnet run --project AntarusPoFinder.App   # запуск из исходников
```

Портативный релизный `.exe` (не требует установленного .NET на целевой машине, весь рантайм
внутри, ~160 МБ):

```bash
dotnet publish AntarusPoFinder.App/AntarusPoFinder.App.csproj -c Release -r win-x64 \
  --self-contained true -p:PublishSingleFile=true -p:IncludeNativeLibrariesForSelfExtract=true \
  -o publish
```

Результат — `publish/AntarusPoFinder.App.exe`. Для полноценного релиза (portable exe + MSI разом,
версия берётся из csproj) используйте `installer/build.ps1` — см. «Обновление приложения» выше.

База данных создаётся автоматически при первом запуске в
`%LocalAppData%\AntarusPOFinder\po_finder.db`.

## Отличия и известные ограничения

По сравнению с предшествующим прототипом на Python сознательно **не портированы**:

- **Второй диск / электросхемы** — просмотр электрических схем шкафов и автодополнение
  названий по второму сетевому диску. Настройка пути в Общих присутствует (зарезервирована),
  сама функция — нет.
- **Легаси-система «Правила»/«Шаблоны»** — более старый способ привязки прошивок к именам
  шкафов без иерархии; полностью заменён иерархической моделью выше.
- **Прямое управление сканером через WIA** — кнопка «Сканировать» использует только упрощённый
  путь (запуск системного приложения «Сканер Windows» + сравнение содержимого папок до/после
  + ручной выбор файла как запасной вариант).

Не реализовано (осознанный бэклог, часть уже подготовлена схемой БД):

- **Уведомления по почте о новых версиях прошивки** — таблица `users`/поле `author_id` у каждой
  версии уже фиксируют, кто и когда загрузил, но рассылки нет.
- **Персональные учётные записи вместо общих ролевых паролей** — вход по AD-логину/паролю
  (см. «Роли и доступ» выше) уже проверяет конкретного человека и запоминает его в собственном
  ростере приложения (Настройки → Пользователи), но результатом всё равно остаётся одна из трёх
  общих ролей (наладчик/программист/администратор), не персональные права по каждому пользователю
  отдельно — вход паролем роли (без AD вовсе) как способ входа никуда не делся и работает
  параллельно.
